## 一、先把 1,244 元的成本差拆开

P2 当前成本 49,888.84 相比 P1 基线 48,644.68 多了 1,244.16 元。先把这个差拆成分项：

| 成本项 | P1 | P2 | 差值 |
|---|---|---|---|
| 固定成本 | 17,200 | 18,400 | **+1,200** |
| 能耗成本 | 25,091.79 | 24,688.13 | -403.66 |
| 碳排成本 | 5,419.37 | 5,327.28 | -92.09 |
| 时间窗惩罚 | 933.53 | 1,473.43 | **+539.90** |
| **总计** | **48,644.68** | **49,888.84** | **+1,244.16** |

**结论非常清楚**：

- 能耗和碳排反而**下降了**（绿区用 EV 替代燃油，正向效果）
- **固定成本多了 1,200 元 = 恰好多开了 3 辆物理车**（P1 用 43 辆，P2 用 46 辆：10 E1 + 1 E2 + 35 F1）
- **时间窗惩罚多了 540 元**，迟到从 4 个增至 12 个，最大迟到从 31.6 分钟升至 124.92 分钟

所以优化方向有两条主线，两条都要同时推进：

**主线 A**：能否减掉那 3 辆多余物理车（节省 1,200 元固定成本）？  
**主线 B**：能否在不增加总成本的前提下，减少时间窗惩罚（540 元）？

两条主线的目标实际上**高度一致**：更紧凑的排班 → 更少物理车 → 更少级联延迟 → 更少惩罚。

## 二、那 3 辆多余车辆从哪来？根因诊断

### 2.1 物理车辆数量的逻辑

P1 和 P2 都是 116 个 trip。P1 用 43 辆跑 116 趟（平均每辆 2.70 趟），P2 用 46 辆跑 116 趟（平均每辆 2.52 趟）。  
**每辆车的平均趟次减少了**，说明 P2 的排班更"稀疏"——同样数量的 trip 被分给了更多物理车。

原因分析：绿色限行政策强制把绿区节点分配给 EV。这在调度层造成了一个**时间碎片化问题**：

```
P1 调度示例（某辆 E1）：
  Trip A：08:00 出发 → 服务绿区客户 → 11:30 返库
  Trip B：11:30 出发 → 服务非绿区客户 → 16:00 返库
  → 1 辆 E1 跑 2 趟，总趟次高效

P2 调度示例（某辆 E1，政策约束下）：
  Trip A：08:00 出发 → 服务绿区客户 → 11:30 返库  ✓ 合规
  Trip B：11:30 出发 → 服务绿区客户（第二批）→ 16:00 返库  ✓ 合规
  → 但如果 Trip B 被分配到了另一辆 E1，就浪费了一辆
```

**真正的问题是**：调度器（scheduler）在给绿区 EV trips 排班时，是否充分利用了"同一辆 E1 可以顺序跑多个绿区 trip"的机会？如果没有，就会多开车。

### 2.2 最大迟到 124.92 分钟的根因

124.92 分钟的迟到**几乎不可能**来自普通路线顺序问题（那顶多导致 20-40 分钟的迟到）。在当前数据下（时间窗平均宽度 72 分钟），124.92 分钟意味着客户在时间窗**关闭将近 2 小时后**才被服务。

最可能的两种原因：

**原因 A（多趟级联）**：某辆物理车的第一趟（Trip X）因绿区政策约束，时间被拉长或出发时间被推后，导致第二趟（Trip Y）大幅延误，Trip Y 内部最后几个客户的到达时间严重超时。

**原因 B（物理不可达）**：某绿区客户的时间窗极紧（比如 48 分钟），但从配送中心到该客户的行驶时间加上前序服务时间，使得在政策约束下根本无法按时到达。这是 Type A（直接不可达）迟到，改算法也没用。

**诊断方法（Codex 应先执行这一步）**：

```python
# 读取 outputs/problem2/default_split/stop_schedule.csv
# 找到到达时间最晚的迟到停靠点，查看：
# 1. 该停靠点所在的物理车辆 ID（vehicle_id）
# 2. 该物理车辆所有 trip 的出发/返回时间（确认是否存在级联）
# 3. 该客户到配送中心的直飞距离（确认是否存在 Type A 不可达）
# 4. 绿区属性（is_green_zone）

# 若是级联：优化调度使第二趟更早出发
# 若是不可达：这是数据约束，接受并在论文中说明
```

### 路线 2：绿区 EV 趟次的"时间分区排班"优化（中等优先级，中等实现成本）

**理由**：当前调度器对 EV 趟次排班的依据主要是"谁有空就分配给谁"，没有针对绿区的时间分区优化。但物理上，绿区服务存在一个关键的时间窗口特征：

```
配送中心 (20,20) 到绿区边界（距原点 10km）的典型路网距离：约 18-25 km
（注：欧氏距离约 13-17 km，路网比值约 1.5）

顺畅时段（9:00-10:00, 55.3 km/h）到达绿区：约 25 min
拥堵时段（8:00-9:00, 9.8 km/h）到达绿区：约 140 min（！！）
```

**关键发现**：如果一辆 EV 在 08:00 出发服务绿区，受拥堵影响约 9:15 才能到达绿区边缘，然后服务一个绿区客户（20 分钟），返回时已经 11:30 左右。这辆 EV 的第二趟如果继续服务绿区，会在 11:30-13:00 的拥堵时段出发，同样低效。

**更优的排班策略**：

```
【最优 EV 绿区趟次排班】
第一趟：09:00 出发（顺畅时段开始）→ ~09:25 到绿区 → 服务 → ~11:00 返库
第二趟：11:00 出发（普通时段）→ ~11:45 到绿区 → 服务 → ~13:00 返库  
第三趟：13:00 出发（顺畅时段）→ ~13:25 到绿区 → 服务 → ~14:45 返库
→ 1 辆 E1 可服务 3 批绿区客户，完全在 16:00 前完成
```

注意：09:00 出发意味着**主动等到顺畅时段**。这在物理上完全合理，且题目没有任何"必须 08:00 出发"的约束。

**实现建议**：在 `scheduler.py` 的 EV 趟次排班逻辑中，对绿区 EV trips 增加一条规则：若该趟次的第一个服务节点是绿区客户，且距配送中心距离 > 20 km，则 `preferred_departure` 设为 `max(available_time, 540)` 而不是 `max(available_time, 480)`（即等到 09:00 顺畅时段再出发）。

这能同时：
- 减少行驶时间和能耗（顺畅 vs 拥堵，55.3 vs 9.8 km/h）
- 让同一辆 E1 更高效地完成多趟绿区服务
- 减少物理车辆数（节省固定成本）

**预期收益**：可能减少 1-2 辆物理车（节省 400-800 元固定成本）并同时减少级联迟到。  
**风险**：中等，需要修改调度器逻辑，需要单元测试验证。

### 路线 3：绿区大需求客户的选择性局部细拆（低-中优先级，中等实现成本）

**对 GREEN_E2_ADAPTIVE 成本过高的诊断**：

GREEN_E2_ADAPTIVE 把所有 19 个绿区服务节点按 E2 容量（1250 kg）重拆，变成 37 个节点（多出 18 个节点），同时因为每个节点对应一趟 trip，趟次数也从 116 增至更多。更多趟次 → 更多物理车 → 更高固定成本，这就是为什么 GREEN_E2_ADAPTIVE 成本高出 7,221 元。

**但 E2 在某些场景下确实有成本优势**：

```python
# 判断某个绿区节点是否适合用 E2 而不是 E1：
# 条件1：节点需求重量 ≤ 1250 kg（E2 能装得下）
# 条件2：节点需求重量 ≤ 1250 kg 且体积 ≤ 8.5 m³
# 条件3：E2 能耗比 E1 低（E2 更轻，U 型曲线下能耗可能更优）

# E2 vs E1 能耗差（轻载、顺畅时段）：
# E1（3000kg，15m³）：EPK = 0.0014*55.3² + ... ≈ 36.19 + 4.27 - 6.64 = 33.82 kWh/100km（空载）
# E2（1250kg，8.5m³）：同样的 EPK 公式，能耗差别取决于载重系数差
# 实际上两种车的 EPK 公式参数相同，能耗差主要来自 load_factor (1 + 0.35*ρ)
# E2 满载（ρ=1）vs E1 半载（ρ=0.5）：1.35 vs 1.175 → E1 半载更省电
```

**结论**：对于需求量 < 1250 kg 的绿区节点，使用 E2 **不一定更省能耗**，因为 E1 空间富余、载重率低、load_factor 小。E2 只在"E1 被占用，而 E2 刚好能装"的情况下有实际价值。

**选择性细拆策略**：只对以下类型的绿区客户局部细拆：

1. 绿区客户总需求 > 1250 kg（E2 装不下，需要 E1，但可以拆成多个 E2 趟次）  
   **→ 这类客户不值得按 E2 细拆，因为 E2 需要多趟且固定成本叠加**

2. 绿区客户总需求 < 1250 kg，且时间窗完全在 8:00-16:00 内（TSF=0）  
   **→ 这类客户可以考虑优先分配给 E2（若 E2 有空余）**

3. 大需求绿区客户被拆分为多个虚拟节点后，虚拟节点 > 1 个的那些  
   **→ 这些本来就已经细拆过了，不需要再拆**

**实际建议**：不要再做新的"全量绿区细拆"变体，而是在现有 148 节点框架下，让调度器的 EV 选型逻辑优先使用 E2 服务轻量绿区节点（需求 ≤ 1250 kg），E1 处理重量绿区节点。这不改变服务节点数，只改变车型分配。

## 四、政策专用算子为什么会抬高成本（根因说明）

当前实验中 `policy_conflict_remove + ev_priority_insert` 组合会抬高成本，原因分析：

1. **`policy_conflict_remove` 的破坏力过强**：它一次性移除整批绿区节点，然后修复算子不一定能找到比之前更好的插入位置，大概率只能等成本，有时候还更差。

2. **`ev_priority_insert` 插入的成本盲区**：它把绿区节点优先插入 EV 路线，但没有考虑"EV 路线已经很满，插入绿区节点会把 EV 的第二趟推到很晚"的级联效应。

3. **更合理的修改方向**：

```python
# 当前错误逻辑：
def ev_priority_insert(node, solution, data):
    # 直接找 EV 路线中成本增量最小的位置插入
    # 问题：没有考虑 EV 路线的时间余量和后续趟次影响
    ...

# 改进逻辑：
def ev_priority_insert_with_cascade_check(node, solution, data, scheduler):
    """
    插入时计算：
    1. 当前 EV 路线的时间预算（还有多少时间余量）
    2. 插入后该 EV 的后续趟次能否按时完成
    3. 如果插入导致后续趟次全部延误超过 X 分钟，则放弃该插入位置
    只有在时间余量充足的 EV 路线中插入
    """
    ...
```

**建议**：暂时不要把政策专用算子作为默认算子，先通过更多迭代和调度优化降低成本，再考虑是否需要这类精细算子。

## 五、关于四层结构重构的客观判断

**问：是否值得重构为 DemandAtom → ServiceVisit → RouteSpec → ScheduledRoute？**

**答：当前阶段不值得，原因如下：**

1. 当前的 `RouteSpec → Route → Solution` 三层结构已经足够表达第二问的所有约束，没有证据表明结构本身是成本瓶颈。

2. 成本差主要来自固定成本（3辆车）和时间窗惩罚（540元），这两者都是**调度质量问题**，不是**数据结构表达问题**。

3. 大重构的风险：破坏现有 32 个测试，引入新的 bug，花费大量时间在工程上而非优化上。

4. **只有一种情况下重构有价值**：当 RouteSpec 的粒度（单个虚拟节点）和 ScheduledRoute 的粒度（物理车辆整日任务）之间缺少一个"已知车辆的分段计划"层，使得 ALNS 无法感知某个节点被插入后对整辆物理车所有后续趟次的影响——这时候插入一个 ServiceVisit 层才有意义。

**最小可行版本（如果要做）**：

```python
# 只新增一个轻量数据类，不改变现有结构
@dataclass
class TripWindow:
    """一辆物理车的一个"可服务窗口"——从上次返库到下次出发之间可用的时间段"""
    vehicle_id: str
    vehicle_type: str
    available_from: float  # 上次返库时间（分钟）
    available_capacity_w: float
    available_capacity_v: float
    
# 在 ALNS 的插入决策中，先查找有足够 TripWindow 的物理车，再插入
# 这比直接插入 RouteSpec 更能预估级联效应
```

这个改动很小（约 50 行），但可能帮助调度器做出更好的排班决策。
