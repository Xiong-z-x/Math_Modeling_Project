# 数据处理模块说明

## 用途

这个文件夹是后续所有求解模块的数据入口。子对话如果只负责行程时间、
成本、初始解或 ALNS，不需要重新阅读全部题面，先读本文件即可快速进入状态。

核心代码：

- `loader.py`：读取、校验、聚合、绿区标记、超需求拆分。
- `__init__.py`：暴露稳定入口 `load_problem_data()`。
- `../data_loader.py`：兼容旧导入路径，内部转发到本文件夹。

推荐导入方式：

```python
from green_logistics.data_processing import load_problem_data

data = load_problem_data(".")
```

## 数据处理流程

1. 读取四个附件文件，支持当前中文文件名：
   - `订单信息.xlsx`
   - `客户坐标信息.xlsx`
   - `距离矩阵.xlsx`
   - `时间窗.xlsx`

2. 标准化列名，避免后续模块受中文列名和 Excel 格式影响：
   - 订单：`order_id, weight, volume, customer_id`
   - 坐标：`node_type, node_id, x_km, y_km`
   - 时间窗：`customer_id, earliest_min, latest_min, window_width_min`

3. 聚合 2169 条订单到客户级需求：
   - `total_weight`
   - `total_volume`
   - `order_count`

4. 排除无订单客户。附件中 98 个客户里，实际有订单客户为 88 个。

5. 按坐标直线距离判断绿色配送区：
   - 圆心是城市中心 `(0, 0)`，不是配送中心 `(20, 20)`。
   - 半径为 `10 km`。
   - 判断公式：`sqrt(x_km^2 + y_km^2) <= 10`。

6. 对超出最大单次配送能力的客户生成虚拟服务节点：
   - 最大拆分基准：`3000 kg` 和 `15 m3`。
   - 拆分数：`ceil(max(total_weight / 3000, total_volume / 15))`。
   - 每个虚拟节点保留 `customer_id`，距离矩阵仍使用原始客户 ID。

## 后续模块最需要注意的坑

1. **虚拟节点 ID 不等于客户 ID。**
   - `service_nodes.node_id` 是算法内部虚拟服务节点。
   - `service_nodes.customer_id` 才是距离矩阵索引。
   - 访问距离时必须用 `node_to_customer[node_id]` 或直接取
     `service_nodes.customer_id`。

2. **绿色区判断不能用距离矩阵。**
   - 距离矩阵的 0 是配送中心，坐标为 `(20, 20)`。
   - 绿色区圆心是城市中心 `(0, 0)`，数据中没有该节点。

3. **无订单客户必须排除。**
   - 时间窗文件包含所有 98 个客户。
   - 求解器只能服务 `customer_demands` 和 `service_nodes` 中的节点。

4. **订单不能作为不可拆原子处理。**
   - 数据里单个订单最大约 `5987 kg`，超过最大车辆载重。
   - 本项目把客户总需求视为可拆配送货量，再生成虚拟服务节点。

5. **时间统一使用从 0:00 起的绝对分钟。**
   - 例如 08:00 为 480，09:09 为 549。
   - 这样能直接匹配题面速度时段，避免“距 8:00 offset”和绝对时刻混用。

6. **拆分只保证不超过最大车 E1 的容量。**
   - 具体插入某条路线时，还必须再次检查车型容量。
   - 例如 F1 容积为 13.5 m3，小于 E1 的 15 m3。

## 当前结论

数据处理部分已形成稳定、可测试入口。后续求解模块应只依赖
`ProblemData` 中的标准化表，不应重新读取 Excel 或自行聚合订单。
