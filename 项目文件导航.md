# 项目文件导航与维护台账

## 0. 文档用途

本文是项目的“文件地图”和“新模型入场指南”。后续任何子对话、新模型或协作者进入项目时，应先阅读本文，再按任务类型阅读对应的主文档或模块 README。

本文需要长期维护。以后凡是新增、移动、删除、重命名关键文件或文件夹，都必须同步更新本文。

---

## 1. 新对话快速进入状态

### 1.1 如果你要理解总路线

按顺序阅读：

1. `项目文件导航.md`：了解当前文件结构和每个文件用途。
2. `解题总思路.md`：理解最终确定的数学建模与算法总路线。
3. `green_logistics/data_processing/README.md`：理解已经完成的数据处理层。
4. `task_plan.md`：查看当前阶段和下一步任务。
5. `progress.md`：查看最近做过什么、验证过什么。

### 1.2 如果你要继续写代码

按顺序阅读：

1. `项目文件导航.md`
2. `解题总思路.md`
3. 当前目标模块的 README 或测试文件
4. `tests/` 中对应测试
5. 目标模块源码

代码实现必须遵守：

- 先写测试，再写实现。
- 不绕开 `green_logistics.data_processing.load_problem_data()` 重新读 Excel。
- 所有参数优先放到 `green_logistics/constants.py`。
- 每次完成后运行对应 pytest。

### 1.3 如果你只负责论文或建模表述

优先阅读：

1. `解题总思路.md`
2. `findings.md`
3. `green_logistics/data_processing/README.md`
4. 原题 PDF 和补充说明 PDF

## 8. 新增文件时的维护规则

新增文件后必须做三件事：

1. 在本文的对应目录台账中增加一行。
2. 如果是关键入口文件，在 `README.md` 中增加入口说明。
3. 如果影响阶段状态，在 `task_plan.md` 和 `progress.md` 中记录。

示例：

```text
新增 green_logistics/travel_time.py 后：
1. 更新本文 3.1 和 4.1；
2. 更新 task_plan.md，将 travel-time 阶段标记为 in_progress 或 complete；
3. 更新 progress.md，记录测试命令和结果；
4. 如有模块 README，则在 README.md 或本文中链接。
```

## 9. 删除或移动文件时的维护规则

删除或移动文件前必须确认：

1. 是否有测试、代码、README、计划文档引用旧路径。
2. 是否需要兼容层保留旧导入路径。
3. 是否会影响后续子对话读取顺序。

删除或移动后必须更新：

- `项目文件导航.md`
- `README.md`
- `task_plan.md`
- `progress.md`
- 相关测试导入路径

如果是代码移动，必须运行对应测试。
