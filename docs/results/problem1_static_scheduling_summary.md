# 第一问静态环境下车辆调度求解总结

## 1. 问题定位

第十八届“华中杯”A题第一问要求在没有绿色限行政策约束的静态环境下，综合考虑异构车队、车辆载重和容积双容量约束、客户软时间窗、时变道路速度、载重相关能耗、碳排放成本和时间窗惩罚成本，建立车辆配送调度模型，并以总配送成本最低为目标输出配送方案。

本项目对第一问的正式目标函数采用题面要求的总配送成本：

$$
\min C = C_{\text{fixed}} + C_{\text{energy}} + C_{\text{carbon}} + C_{\text{tw}}.
$$

其中时间窗是软约束，早到和迟到通过成本计入目标函数，而不是作为必须为零的硬约束。因此，最终解不以“零迟到”为硬目标；服务质量指标用于诊断和解释，不能替代官方总成本目标。

## 2. 数据事实与易错点

第一问求解严格使用数据处理层的稳定入口：

```python
from green_logistics.data_processing import load_problem_data
```

不重新读取或重新聚合 Excel。当前数据处理结果如下：

| 指标 | 数值 |
| --- | ---: |
| 原始订单数 | 2169 |
| 坐标节点数 | 99，含配送中心和 98 个客户 |
| 有订单客户数 | 88 |
| 拆分后虚拟服务节点数 | 148 |
| 总重量需求 | 285122.64682 kg |
| 总体积需求 | 772.431 m3 |
| 有订单绿色区客户数 | 12 |
| 绿色区虚拟服务节点数 | 19 |

必须固定以下事实，防止后续建模幻觉：

1. 虚拟服务节点 `node_id` 不等于原始客户编号。距离矩阵索引必须使用 `customer_id`。
2. 配送中心是距离矩阵节点 `0`，坐标文件记录为 `(20,20)`；绿色配送区圆心是城市中心 `(0,0)`，二者不是同一点。
3. 补充说明给出绿色区为以 `(0,0)` 为圆心、半径 `10 km` 的圆形区域。第二问使用该定义，不能把配送中心当作绿色区圆心。
4. 时间统一使用从 `0:00` 起的绝对分钟数，`08:00=480`。
5. 补充说明只给出 `08:00-17:00` 的速度分布。本实现默认 `17:00` 后沿用普通时段速度；这是一项必须在论文中说明的延拓假设。
6. 题面没有给出 `22:00` 返库硬约束。本实现不把 `22:00` 写成默认硬约束，只把返库时间作为诊断或情景分析参数。

## 3. 模型假设

为使题目数据、车辆容量和算法实现保持物理一致，第一问采用如下假设：

1. 客户总需求可以拆分为多个虚拟服务节点。原因是数据中存在单个原始订单重量超过最大车辆载重的情况，如果坚持订单不可拆，会导致数据本身不可行。
2. 每条 `Route` 表示一次从配送中心出发并返回配送中心的 depot-to-depot 趟次。物理车辆可在完成一趟后继续执行同车型后续趟次，固定成本按实际启用的物理车辆计一次。
3. 车辆到达客户后若早于时间窗下界，则等待至下界后服务；若晚于上界，则产生迟到惩罚。服务时间固定为 `20 min`。
4. 行驶时间按时段速度分段积分。跨时段路段不能使用“出发时速度 × 全程距离”的近似。
5. 能耗公式中的速度是随机变量。由于题面能耗函数含二次项，计算期望能耗时使用 Jensen 修正：

   $$
   E[v^2] = \mu^2 + \sigma^2.
   $$

6. 载重影响能耗。车辆出发时载重最高，沿途卸货后逐步下降，弧段能耗按服务前剩余载重修正。

## 5. 成本模型

### 5.1 固定成本

固定成本按启用物理车辆数计算，而不是按趟次数计算：

$$
C_{\text{fixed}} = f \sum_{k\in K} n^{\text{used}}_k.
$$

其中 \(f=400\) 元。当前正式结果启用 43 辆物理车，因此：

$$
C_{\text{fixed}} = 43 \times 400 = 17200.
$$

### 5.2 时变行驶时间

补充说明给出的速度分布为：

| 时段 | 状态 | 分布 |
| --- | --- | --- |
| 08:00-09:00 | 拥堵 | \(N(9.8,4.7^2)\) |
| 09:00-10:00 | 畅通 | \(N(55.3,0.1^2)\) |
| 10:00-11:30 | 普通 | \(N(35.4,5.2^2)\) |
| 11:30-13:00 | 拥堵 | \(N(9.8,4.7^2)\) |
| 13:00-15:00 | 畅通 | \(N(55.3,0.1^2)\) |
| 15:00-17:00 | 普通 | \(N(35.4,5.2^2)\) |
| 17:00-24:00 | 延拓普通 | \(N(35.4,5.2^2)\) |

若车辆在时刻 \(t\) 从 \(i\) 行驶到 \(j\)，距离 \(d_{ij}\) 可能跨越多个速度时段。设跨越的子段集合为 \(P_{ij}(t)\)，每个子段距离为 \(\Delta d_p\)，速度均值为 \(\mu_p\)，则行驶时间为：

$$
\tau_{ij}(t) = \sum_{p\in P_{ij}(t)} \frac{60\Delta d_p}{\mu_p}.
$$

到达时间递推为：

$$
a_j = t_i + \tau_{ij}(t_i),
$$

其中 \(t_i\) 是上一节点离开时间。

### 5.3 软时间窗惩罚

客户 \(i\) 的时间窗为 \([e_i,l_i]\)。车辆到达时间为 \(a_i\)，则等待和迟到分别为：

$$
W_i = \max(e_i-a_i,0),
$$

$$
L_i = \max(a_i-l_i,0).
$$

题面给定早到等待成本为 `20 元/小时`，迟到成本为 `50 元/小时`，因此：

$$
C_{\text{tw}}=\sum_{i\in S}\left(\frac{20}{60}W_i+\frac{50}{60}L_i\right).
$$

服务开始和离开时间为：

$$
b_i = \max(a_i,e_i),
$$

$$
t_i = b_i + s,\quad s=20.
$$

### 5.4 能耗成本

燃油车和电动车的空载基准能耗函数分别为：

$$
g_F(v)=0.0025v^2-0.2554v+31.75,
$$

$$
g_E(v)=0.0014v^2-0.12v+36.19.
$$

由于速度服从正态分布，二次项不能直接使用 \(\mu_p^2\)，而应使用：

$$
E[g_F(v)] = 0.0025(\mu_p^2+\sigma_p^2)-0.2554\mu_p+31.75,
$$

$$
E[g_E(v)] = 0.0014(\mu_p^2+\sigma_p^2)-0.12\mu_p+36.19.
$$

设弧段行驶前车辆剩余载重为 \(w\)，车型容量为 \(Q_k\)，载重比例为：

$$
\lambda = \frac{w}{Q_k}.
$$

燃油车载重修正系数取 \(1+0.40\lambda\)，电动车取 \(1+0.35\lambda\)。对跨时段弧段 \((i,j)\)，燃油消耗和电耗分别为：

$$
E^F_{ij} = \left(1+0.40\lambda\right)
\sum_{p\in P_{ij}(t)}
\frac{\Delta d_p}{100}E[g_F(v_p)],
$$

$$
E^E_{ij} = \left(1+0.35\lambda\right)
\sum_{p\in P_{ij}(t)}
\frac{\Delta d_p}{100}E[g_E(v_p)].
$$

燃油价格为 `7.61 元/L`，电价为 `1.64 元/kWh`，所以：

$$
C_{\text{energy}} =
7.61\sum E^F_{ij} + 1.64\sum E^E_{ij}.
$$

### 5.5 碳排成本

燃油车碳排因子为 `2.547 kg/L`，电动车电力碳排因子为 `0.501 kg/kWh`，碳价为 `0.65 元/kg`。因此：

$$
G = 2.547\sum E^F_{ij}+0.501\sum E^E_{ij},
$$

$$
C_{\text{carbon}} = 0.65G.
$$

## 6. 约束条件

### 6.1 服务覆盖约束

每个虚拟服务节点必须且只需服务一次：

$$
\sum_{r\in R}\sum_{j} x^r_{ij}=1,\quad \forall i\in S.
$$

当前正式输出中，148 个虚拟服务节点均被服务一次，无缺失和重复。

### 6.2 车辆容量约束

每条趟次的总重量和总体积不能超过对应车型容量：

$$
\sum_{i\in r} q_i \le Q_k,\quad
\sum_{i\in r} u_i \le V_k.
$$

当前正式输出中所有趟次均满足容量约束。

### 6.3 车辆数量约束

每种车型启用的物理车辆数不能超过题面给定数量：

$$
n^{\text{used}}_k \le N_k,\quad \forall k\in K.
$$

正式结果启用 `E1: 10`、`F1: 33`，均不超过车辆上限。

### 6.4 趟次连续性约束

每条趟次从配送中心出发并返回配送中心：

$$
0 \rightarrow i_1 \rightarrow i_2 \rightarrow \cdots \rightarrow i_m \rightarrow 0.
$$

物理车辆执行多趟时，后一趟出发时间不得早于前一趟返回时间。当前模型默认无额外装卸重装时间；若后续题目或论文需要，可通过 `SchedulingConfig.reload_time_min` 做情景分析。

### 6.5 时间递推约束

到达、等待、服务和离开时间必须满足：

$$
a_{i_h}=t_{i_{h-1}}+\tau_{i_{h-1},i_h}(t_{i_{h-1}}),
$$

$$
b_{i_h}=\max(a_{i_h},e_{i_h}),
$$

$$
t_{i_h}=b_{i_h}+20.
$$

该递推保证路线评价符合行驶和服务的物理逻辑。

## 7. 求解算法

第一问采用“可行初解构造 + 物理车辆排班 + ALNS 优化 + 诊断修正”的启发式框架。选择启发式而非精确 MILP 的原因是当前问题包含 148 个虚拟服务节点、异构车队、软时间窗、时变速度、载重相关能耗和多趟物理车复用，直接精确建模规模较大，不适合在建模竞赛时间内稳定求解。

### 7.1 数据层

数据层完成如下任务：

1. 标准化四个 Excel 附件；
2. 聚合 2169 条订单到客户级需求；
3. 排除无订单客户；
4. 按 `(3000 kg, 15 m3)` 最大单趟容量拆分客户需求，生成 148 个虚拟服务节点；
5. 保留 `node_id -> customer_id` 映射，保证距离查询使用原始客户编号。

### 7.2 初始解

初始解先构造未排班的 `RouteSpec`：

1. 按客户时间窗和需求规模排序；
2. 在重量和体积容量允许的前提下尝试贪心插入；
3. 对插入后的路线完整重算时间、能耗、碳排和时间窗成本；
4. 若现有路线无法合理插入，则开启新趟次。

随后由 `schedule_route_specs()` 将趟次分配给物理车辆。排班时既考虑车辆是否可用，也考虑服务质量风险，避免把多个早截止时间窗的趟次机械叠加到同一物理车上。

### 7.3 ALNS 搜索

ALNS 在 `RouteSpec` 层进行节点删除和再插入，每次候选解都会重新排班并完整评价。

破坏算子包括：

| 算子 | 作用 |
| --- | --- |
| Random Remove | 随机移除若干服务节点，增加搜索多样性 |
| Worst Cost Remove | 移除边际成本较高节点 |
| Related Remove | 移除空间、时间或需求相近节点，便于重组 |
| Time Penalty Remove | 针对时间窗惩罚较高节点 |
| Actual Late Remove | 针对真实排班后的迟到节点 |
| Late Suffix Remove | 移除迟到节点之后的后缀，修复局部时间递推 |
| Midnight Route Remove | 避免极端晚归路线 |
| Late Route Split | 对迟到严重路线做拆分尝试 |

修复算子包括：

| 算子 | 作用 |
| --- | --- |
| Greedy Insert | 选择最小增量成本插入 |
| Regret-2 Insert | 优先插入选择余地小的节点 |
| Time-Oriented Insert | 优先考虑时间窗压力 |

接受准则使用模拟退火思想，帮助跳出局部最优。需要特别说明的是：搜索过程中使用的 `search_score` 只用于引导探索和减少极端迟到，正式最优解选择仍以题面总成本 `total_cost` 为第一标准。

### 7.4 诊断与纠偏

第一问优化过程中曾出现“为追求少用车而导致系统性迟到”的现象。项目最终采取的纠偏原则是：

1. 不把零迟到误当作题面硬目标；
2. 不用 22:00 返库等题面未给出的硬约束强行改写问题；
3. 保留软时间窗成本，并按总成本选择正式答案；
4. 用迟到停靠数、最大迟到、跨午夜返库等指标作为服务质量诊断；
5. 对残余迟到输出分类，解释其来源。

当前 4 个残余迟到停靠的诊断如下：

| 类型 | 数量 | 解释 |
| --- | ---: | --- |
| Type A direct-infeasible | 1 | 即使 08:00 从配送中心直达也会迟到，接近物理不可避免或数据时间窗过紧 |
| Type B multi-trip cascade | 2 | 直达不迟到，但由于物理车执行后续趟次产生级联延迟 |
| Type C route-order/local-optimum | 1 | 与路线内部顺序或局部最优有关 |

该诊断说明残余迟到不是被忽略的问题，而是软时间窗成本权衡后的结果。进一步追求零迟到可以实现，但在已测试方案中会提高官方总成本，因此不作为第一问正式答案。

## 12. 第二问接口准备

虽然第一问不考虑绿色限行政策，但当前代码已经为第二问预留接口：

1. `green_logistics/policies.py` 中有 `NoPolicyEvaluator` 和 `GreenZonePolicyEvaluator` 框架；
2. `green_logistics/diagnostics.py` 可输出绿色区容量诊断和政策冲突预判；
3. `green_logistics/trips.py` 中的 `TripDescriptor` 记录趟次是否触达绿色区；
4. `green_logistics/scheduler.py` 中的 `SchedulingConfig` 支持情景参数，如返库时间限制、重装时间和出发时间网格搜索。

当前第一问正式方案在第二问“8:00-16:00 燃油车限制进入或服务绿色区”政策下存在 `12` 个潜在冲突停靠。因此第二问不能简单沿用第一问路线，需要在电动车分配、绿色区服务时间和燃油车路径约束上重新优化。

## 13. 可解释性与创新点

第一问方案的主要可解释性和创新点包括：

1. 使用虚拟服务节点处理超容量客户需求，同时保留原始客户 ID 作为距离矩阵索引，解决了数据可行性和路径计算一致性问题。
2. 将行驶时间和能耗都按速度时段分段积分，避免跨时段路线被粗糙近似。
3. 对二次能耗函数使用 Jensen 修正，使随机速度下的期望能耗计算更符合数学期望。
4. 在同一模型内同时处理重量、体积、软时间窗、载重能耗、碳排成本和物理车辆多趟复用。
5. 将官方总成本目标和服务质量诊断分离：正式答案按总成本选择，诊断指标用于发现不合理迟到和晚归，而不是改写题意。
6. 采用 C-lite 架构，把趟次构造和物理车辆排班适度解耦，为第二问绿色限行和第三问动态调度保留接口。

## 14. 结论

在无绿色限行政策条件下，第一问最终得到的静态配送方案总成本为 `48644.68 元`，启用 `43` 辆物理车辆，完成 `116` 个 depot-to-depot 趟次，服务全部 `148` 个虚拟服务节点，并满足所有容量和车辆数量约束。方案总行驶距离为 `13384.29 km`，碳排放量为 `8337.49 kg`，时间窗惩罚成本为 `933.53 元`。

该结果没有追求表面上的零迟到，而是在题面“总配送成本最低”的目标下保留软时间窗罚项，并通过诊断确认残余迟到规模较小且来源明确。第一问至此可作为后续第二问绿色限行优化和第三问动态调度的基准方案。
